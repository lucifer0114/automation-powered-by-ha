# v1.0.4 Speed Optimization Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Shorten end-to-end Weibo evidence generation time without relaxing the current accepted quality bar: full post body, minimal unrelated comments, exact red-box placement, and stable like/comment/capture behavior.

**Architecture:** Keep `script/weibo_manual_comment_flow.py` as the single entry point and optimize only the slow path internals: readiness waits, submit/capture branching, and comment-location retries. Add timing instrumentation first, then tighten waits only where the evidence shows they are over-conservative. Preserve the current tight contextual screenshot composition and current stable selectors unless tests or telemetry justify change.

**Tech Stack:** Python 3.12, Playwright sync API, Pillow, pytest

---

## Non-negotiable constraints

- Do **not** optimize by choosing easier posts; future runs will usually receive a user-supplied Weibo URL.
- Do **not** relax correctness checks just to save time.
- The first screenshot may be delivered immediately **only if** it is already clear and the red box is exact.
- Preserve the current accepted workflow shape: like → comment → tight screenshot → exact box.

---

## Current baseline to optimize

Observed hot spots in `script/weibo_manual_comment_flow.py`:
- `wait_for_page_ready(...)` uses `networkidle` first, then broad selector fallback.
- `wait_for_time_sort_refresh(...)` sleeps in fixed checkpoints: `350`, `800`, `1400` ms.
- submit flow waits `500` ms after opening the composer and `1200` ms after clicking submit.
- `locate_comment(...)` retries through multiple scroll/sort branches even when the desired signal may already be detectable earlier.
- `capture_comment_artifacts(...)` always takes a full-page base screenshot before generating the tight contextual crop.

These are the areas to optimize first because they add deterministic latency to every run.

## Task 2: Replace fixed submit sleeps with readiness-based waits

**Objective:** Remove avoidable fixed waits after opening the composer and after clicking submit.

**Files:**
- Modify: `script/weibo_manual_comment_flow.py:756-780`
- Test: `tests/test_reliability_regressions.py`
- Test: `tests/test_wait_strategy.py`

**Step 1: Add a dedicated composer-ready helper**
- Create a helper such as `wait_for_comment_composer_ready(page, timeout_ms=1200)`.
- It should prefer specific composer selectors instead of blind `page.wait_for_timeout(500)`.
- Candidate selectors:
  - `[data-testid="comment-composer"] textarea`
  - `[data-testid="comment-composer"] [contenteditable="true"]`
  - `textarea[placeholder="发布你的评论"]`

**Step 2: Add a dedicated post-submit helper**
- Create a helper such as `wait_for_after_submit_signal(page, comment_text)`.
- Prefer a fast visible signal over unconditional `page.wait_for_timeout(1200)`.
- Candidate signals:
  - comment list visible
  - matching comment text visible
  - known comment container visible after refresh

**Step 3: Keep a bounded fallback**
- If no fast signal appears, keep a short fallback wait so the flow remains robust.
- The fallback should be shorter and more targeted than the current unconditional 1200 ms.

**Step 4: Add tests**
- Test that the composer helper returns immediately when the targeted selector is already visible.
- Test that the post-submit helper falls back safely when no fast signal appears.

**Step 5: Verify**
Run:
```bash

```
Expected: pass.

**Step 6: Commit**

## Task 3: Short-circuit comment location earlier after sort switch

**Objective:** Reduce the time spent in `locate_comment(...)` when the target comment becomes visible soon after switching to `按时间`.

**Files:**
- Modify: `script/weibo_manual_comment_flow.py:380-442`
- Test: `tests/test_p0_regressions.py`

**Step 1: Make `wait_for_time_sort_refresh(...)` progressively check sooner**
- Keep the function, but make it return as soon as a positive signal appears.
- Replace the current coarse checkpoint strategy with an early-first approach, for example:
  - shorter first probe
  - one medium probe
  - one final fallback probe
- Preserve bounded total wait time.

**Step 2: Avoid unnecessary follow-up scroll branches when sort already worked**
- If `wait_for_time_sort_refresh(...)` positively detects the comment, skip the later scroll-and-retry branches.
- Keep the later branches only for true misses.

**Step 3: Record which branch won**
- Add a lightweight field or print line showing whether location succeeded via:
  - direct
  - after-time-sort-fast
  - after-time-sort-final
  - after-scroll
- This will help future tuning.

**Step 4: Add tests**
- Add a regression test proving that a positive post-sort signal returns before extra scroll retries.
- Add a regression test proving fallback branches still run when needed.

**Step 5: Verify**
Run:
```bash
pytest -q tests/test_p0_regressions.py
```
Expected: pass.

**Step 6: Commit**
